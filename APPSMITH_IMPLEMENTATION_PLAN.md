# AppSmith Reimplementation Plan

## Goal

Reimplement the current Metabase + selected n8n UI/action surface in AppSmith,
while keeping n8n for workflows where orchestration, scheduling, external
integration, or document generation still adds value.

This is not a full "replace n8n" plan.

The target split is:

- `AppSmith` for operator-facing pages, search, detail views, admin screens,
  parameterized actions, and direct database-backed CRUD where appropriate
- `n8n` for scheduled jobs, email parsing, batch processing, PDF generation,
  sync tasks, and any multi-step workflow with retries/state

## Desired End State

One internal admin application in AppSmith that replaces the current Metabase
dashboard-driven operations flow.

It should cover:

- membership search
- member detail views
- signup batch views
- batch member rows
- manual batch item entry
- no-address email controls
- raw-data audit/review pages
- global settings editing
- operational/admin review cards and tables

And it should integrate with n8n in two ways:

- direct webhook invocation where n8n is still the right execution engine
- optional direct AppSmith queries/JS for simple actions that do not need n8n

## What Should Move To AppSmith

### High-value UI screens

These are currently awkward in Metabase because they need richer interaction or
stateful navigation:

- `Contacts - Selected Year` / search results
- member detail page
- signup batch actions page
- signup batch member rows
- missing-signup review list
- ambiguous/weak raw-match review pages
- global settings admin page

### Action flows that are better as AppSmith forms/buttons

- manual batch item form
- signup batch actions launcher
- "create missing signup capture" action
- conditional action enable/disable logic
- richer confirmation dialogs and input validation

### Simple read/write admin pages

- `global_settings`
- `email_templates`
- possibly `images` metadata management

## What Should Stay In n8n

These still fit n8n better than AppSmith:

- ClubSpark auth and export workflows
- new member email parser
- sync local source of truth to cloud
- scheduled membership snapshot capture
- label PDF generation
- DL envelope PDF generation
- no-address batch email send workflow
- any workflow that needs retries, branching, external service coordination, or
  scheduled execution

## Architecture

### Data access

AppSmith should connect directly to the same Postgres databases used today.

Recommended data sources:

- local Postgres
- cloud Postgres if you need a cloud-admin deployment too

### Workflow integration

Use AppSmith as the action initiator when needed:

- call n8n webhooks for workflow-backed actions
- call SQL directly for simple mutations where no workflow engine is required

### Shared contract boundary

Preserve existing Postgres views/functions as the system contract where they are
already stable and useful.

Examples:

- `vw_signup_batches_summary`
- `vw_signup_batch_consolidated`
- `vw_signup_batch_items`
- `resolve_best_contact_row(...)`
- match-review views

This reduces migration risk and avoids rewriting good SQL into AppSmith JS.

## Migration Principles

### 1. Rebuild screens, not underlying business logic

Do not start by rewriting Postgres logic.

Prefer:

- reusing existing views/functions
- reusing existing n8n workflows
- swapping only the UI/action shell first

### 2. Move actions selectively

A button does not automatically belong in n8n.

Use AppSmith directly when the action is:

- a single SQL mutation
- a simple parameterized query
- a read-only navigation/detail flow

Use n8n when the action is:

- multi-step
- side-effect heavy
- external-system dependent
- scheduled
- retriable/auditable as a workflow

### 3. Keep a thin adapter layer

Do not let AppSmith pages each invent their own logic.

Create stable query/function entry points for:

- member search
- member detail
- batch list
- batch detail/member rows
- missing-signup review
- global settings edit

## Scope Breakdown

## Phase 1: Discovery And Surface Mapping

Create a full inventory of what the current Metabase + webhook UI actually does.

### Deliverables

- screen inventory
- card-to-screen mapping
- workflow/webhook dependency map
- "move to AppSmith" vs "keep in n8n" decision list

### Specific items to map

- dashboard `11` tabs and cards
- search-result pages
- detail pages currently generated by webhook HTML
- batch actions and related links
- admin/database review screens

## Phase 2: Define The AppShell And Navigation

Design the AppSmith application structure.

### Suggested pages

- `Search`
- `Member Detail`
- `Signup Batches`
- `Batch Detail`
- `Manual Batch Item`
- `Missing Signup Capture`
- `Database Review`
- `Settings`
- `Templates`

### Deliverables

- page map
- navigation structure
- parameter-passing approach between pages

## Phase 3: Stabilize Backend Query Contracts

Before building pages, define the exact SQL entry points AppSmith will use.

### Recommended first query contracts

- member search query
- member detail query by `raw_member_id`
- batch summary query
- batch member rows query by `batch_id`
- missing-signup query
- match-review queries
- global settings read/update queries

### Recommended backend changes

- favor row-id-based navigation over name-based links
- keep helper IDs hidden in UI but present in query results
- create dedicated views/functions where current SQL is too embedded in cards

## Phase 4: Rebuild The Search Flow In AppSmith

This should be the first full migrated user flow.

### Why first

- high operational value
- currently stretched across Metabase cards and webhook HTML
- benefits immediately from proper navigation and forms

### AppSmith features to implement

- search box with year filter
- results table
- `Add Items` action button
- per-row signup status
- row click to member detail page
- hidden technical IDs

### Backend expectation

- one search query returning:
  - `raw_member_id`
  - visible member fields
  - signup status
  - navigation/action context

## Phase 5: Rebuild Member Detail And Manual Actions

Replace webhook-generated HTML pages with native AppSmith pages/forms.

### Member Detail page

Show:

- selected member identity
- resolved membership rows
- matched signup rows
- matched contact rows
- best contact/address summary

### Manual Batch Item page/modal

Input:

- batch id
- regular tags
- parent tags
- keys
- notes

Execution options:

- direct SQL insert if the logic is simple and safe to centralize in DB
- or keep n8n webhook if you want workflow-level audit/behavior unchanged

### Recommendation

If `add-manual-batch-item` remains non-trivial, keep the workflow in n8n and
have AppSmith call it.

## Phase 6: Rebuild Signup Batch Operations

Replace the current Metabase batch table + webhook HTML action page with proper
AppSmith pages.

### Signup Batches page

Include:

- batch list
- counts and totals
- status
- consolidated item totals
- one actions column with buttons

### Batch Detail page

Include:

- member rows in batch
- grouped/consolidated view
- no-address email sent status
- direct action controls

### Actions

Decide per action:

- `Labels`
  - probably keep in n8n
- `Envelopes`
  - probably keep in n8n
- `No Address Emails`
  - keep in n8n
- `Create batch`, `Complete batch`
  - evaluate whether direct SQL + function is enough, otherwise keep in n8n

## Phase 7: Rebuild Admin / Database Review

Move the current review/admin cards into AppSmith pages with better drilldown.

### Candidate screens

- ambiguous raw matches
- weak name-only matches
- latest raw match outcomes
- latest raw match audit
- global settings editor

### Benefits over Metabase

- better filtering
- richer row actions
- forms and mutation support
- less dependence on card visualization settings

## Phase 8: Settings And Template Management

AppSmith is a better home than Metabase for editable operational settings.

### Recommended first editable tables

- `global_settings`
- `email_templates`

### Optional later

- `images`
- batch/email config tables if introduced later

### Guardrails

- read/write separation where needed
- simple validation in UI
- audit metadata if edits become operationally significant

## Phase 9: Authentication And Access Model

Decide how AppSmith access will work.

### Questions to answer

- internal-only app or externally reachable through Pangolin
- SSO or local auth
- edit vs read-only roles
- whether cloud AppSmith is needed, or local-only is sufficient

### Recommendation

Start with internal admin access only.

Do not solve public/member access in the first migration.

## Phase 10: Cutover Strategy

Do not big-bang replace Metabase.

### Safe cutover order

1. build AppSmith search flow
2. build AppSmith member detail/manual actions
3. build AppSmith signup batch pages
4. build AppSmith admin/settings pages
5. switch operators to AppSmith for those flows
6. retire equivalent Metabase cards/pages once usage is proven

### During transition

Keep:

- Metabase for existing reporting/analytics
- AppSmith for operational workflows

That split is healthy and reduces migration pressure.

## Recommended Backend Refactors Before Or During Migration

These will make AppSmith implementation safer:

- create dedicated SQL views/functions for:
  - member detail by `raw_member_id`
  - search result set
  - batch member rows
  - missing-signup candidates
- keep row-id-based navigation everywhere possible
- add typed query contracts around the shared SQL used by AppSmith
- centralize mutation logic in DB functions when the same action may be called
  from AppSmith and n8n

## Risks

### Risk: Rewriting too much logic in AppSmith JS

Mitigation:

- keep business logic in Postgres views/functions or n8n where it already works

### Risk: Replacing Metabase reporting unnecessarily

Mitigation:

- migrate operational UI first
- keep Metabase for analytics/reporting until there is a clear reason to move it

### Risk: AppSmith pages coupling directly to unstable raw-table shapes

Mitigation:

- use dedicated views/functions
- avoid direct page queries over unstable raw tables when possible

### Risk: Mixed action paths becoming inconsistent

Mitigation:

- define clearly which actions belong to AppSmith direct SQL vs n8n workflow

## Suggested First Slice

Do this first:

1. AppSmith `Search` page
2. AppSmith `Member Detail` page
3. AppSmith `Manual Batch Item` action

Why:

- it replaces the most awkward current combination of Metabase tables and
  webhook-generated HTML
- it already benefits from row-id-based identity work completed recently
- it proves the AppSmith + Postgres + n8n split before the larger batch/admin
  migration

## Concrete Next-Step Plan When Ready

When you decide to start, the first implementation plan should be:

1. inventory current operational Metabase cards and webhook pages
2. define the AppSmith page map
3. extract or stabilize the SQL contracts for search and member detail
4. build the Search page
5. build the Member Detail page
6. wire the Manual Batch Item action
7. run operator acceptance on the David Smith / duplicate-name cases

## Non-Goals For The First Iteration

- replacing all Metabase analytics/reporting
- replacing all n8n workflows
- replacing PDF generation
- replacing scheduled jobs
- building a public-facing member portal
