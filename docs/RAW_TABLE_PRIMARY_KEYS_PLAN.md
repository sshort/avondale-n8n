## Goal

Add durable primary key `id` columns to:

- `public.raw_members`
- `public.raw_contacts`

Requirements:

- IDs must be unique.
- IDs must be preserved across future imports.
- IDs must replicate unchanged to cloud.
- Local remains the source of truth.

## Summary

This cannot be done safely by only adding `bigserial` columns to the existing full-refresh tables.

If imports continue to truncate and reload `raw_members` / `raw_contacts`, the new IDs will churn on every import and will not be useful as durable identifiers.

The safe implementation is:

1. Add durable IDs and lifecycle fields to the current tables.
2. Load ClubSpark files into staging tables.
3. Reconcile staging rows onto existing current rows using stable matching rules.
4. Insert only genuinely new rows into the current tables.
5. Mark missing rows as non-current or archive them instead of recreating everything.
6. Replicate those same IDs to cloud with explicit upsert logic.

## Schema Changes

### 1. Add primary keys locally

Add to both `public.raw_members` and `public.raw_contacts`:

- `id bigint generated by default as identity primary key`

### 2. Add lifecycle fields locally

Add to both tables:

- `first_seen_at timestamptz not null default now()`
- `last_seen_at timestamptz not null default now()`
- `is_current boolean not null default true`
- optional: `source_hash text`

Purpose:

- preserve row identity without recreating rows
- track when a row first appeared
- track whether a row is still present in current source data

### 3. Add staging tables

Create:

- `public.raw_members_import_staging`
- `public.raw_contacts_import_staging`

These should mirror the incoming file shape and should not use durable business IDs.

### 4. Optional unique indexes

Only add these after checking duplicates:

- unique normalized `Venue ID`
- unique normalized `British Tennis Number`

Do not assume those are complete or clean.

## Matching Strategy

### raw_members

Reconcile imported rows onto current rows in this order:

1. match by `Venue ID`
2. else match by `British Tennis Number`
3. else match by exact normalized `First name + Last name + Membership Category`
4. else match by exact normalized `First name + Last name + Membership`
5. otherwise insert a new row

Notes:

- do not use email as the main identity key
- family/shared emails make that unsafe

### raw_contacts

Reconcile imported rows onto current rows in this order:

1. match by `Venue ID` if present
2. else match by exact normalized `First name + Last name`
3. prefer the row with better address/email completeness where duplicates exist
4. otherwise insert a new row

## Import Flow Changes

Current truncate-and-reload behavior must be replaced.

For each import:

1. load CSV into staging table
2. normalize staging values
   - trim whitespace
   - convert empty strings to `NULL`
   - uppercase postcodes
   - normalize `Venue ID` / BTN text
3. update matched current rows in place
   - preserve existing `id`
   - set `last_seen_at = now()`
   - set `is_current = true`
4. insert unmatched staging rows into current table
   - let local generate new `id`
   - set `first_seen_at = now()`
   - set `last_seen_at = now()`
5. rows not seen in the import should be:
   - marked `is_current = false`
   - or archived
   - but not destroyed and recreated

## Cloud Replication Changes

Cloud must mirror local IDs exactly.

### Required changes

1. update sync workflow to use explicit column lists including `id`
2. cloud writes must use:
   - `INSERT ... ON CONFLICT (id) DO UPDATE`
3. after sync, run `setval(...)` on the cloud sequences for:
   - `raw_members_id_seq`
   - `raw_contacts_id_seq`
4. do not allow cloud-side imports to generate independent IDs for these tables

## Workflow and SQL Changes Needed

### n8n workflows

1. `clubspark-contacts-export`
   - stop loading directly into `raw_contacts`
   - load staging, then reconcile

2. `clubspark-members-export`
   - stop loading directly into `raw_members`
   - load staging, then reconcile

3. `sync-raw-tables-to-cloud`
   - replace `SELECT *` from `raw_contacts` / `raw_members`
   - use explicit column lists including `id`
   - write to cloud with `ON CONFLICT (id)`
   - reset cloud sequences after sync

### SQL / helper functions

1. `prepare_raw_contacts_import()`
   - likely needs to become staging-table prep, not current-table reset

2. `prepare_raw_members_import()`
   - likely needs the same treatment

3. `archive-raw-members-by-season`
   - must stop using `RETURNING rm.*`
   - use explicit column lists instead
   - decide whether archive tables should store current raw row IDs or their own archive IDs

## Metabase Impact

Most dashboard logic should survive unchanged because it uses named columns, not whole-row copies.

### Likely no logic changes required

- `Members - Selected Year`
- `Members by category`
- `Age Ranges`
- most list/count cards based on `raw_members`

### Likely cleanup required

- refresh metadata after schema change
- hide `raw_members.id` and `raw_contacts.id` on user-facing cards
- admin/raw table cards may show an extra column unless explicitly hidden

### Important principle

The new IDs should not become the person-matching key for the dashboard logic unless the import preservation work is complete.

## Migration Sequence

1. take local and cloud backups
2. add `id` and lifecycle fields to local current tables
3. backfill IDs on existing local rows
4. add local staging tables
5. implement local reconcile SQL/functions
6. change local contacts import workflow to use staging + reconcile
7. change local members import workflow to use staging + reconcile
8. run the same import twice and verify IDs do not change
9. add the same `id` columns to cloud current tables
10. update sync workflow to replicate IDs and reset cloud sequences
11. run sync to cloud
12. refresh local and cloud Metabase metadata
13. only then remove the old full-replace assumptions

## Verification Checklist

### Local verification

1. import the same contacts file twice
   - row count unchanged
   - IDs unchanged

2. import the same members file twice
   - row count unchanged
   - IDs unchanged

3. modify one source row
   - same ID
   - changed attributes updated

4. remove one source row
   - row not recreated
   - row marked non-current or archived

### Cloud verification

1. sync local to cloud
   - same IDs on both sides

2. rerun sync
   - no duplicate rows
   - no ID churn

3. check cloud sequences
   - next generated ID is above current max

## Rollback

Keep rollback simple:

1. take pre-migration dumps
2. do not drop old archive/history structures during rollout
3. keep new reconcile imports isolated until verified
4. if needed, restore dumps and revert workflow versions

## Recommendation

If this work is implemented, the right end state is:

- local assigns IDs once
- imports preserve those IDs
- cloud mirrors those IDs
- Metabase treats them as technical row IDs, not business identity keys

Without the staging + reconcile step, adding auto-increment IDs to `raw_members` and `raw_contacts` is not enough.
